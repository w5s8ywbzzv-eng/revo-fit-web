# revo — 申請までの進め方（ロードマップ）

前提：**開発マシンは Windows のみ（Mac なし）**。Apple Developer Program は登録済み。
このドキュメントは「revo を App Store / Google Play に出す」までの道筋を、Windows で完結できる形で整理したものです。

---

## 0. 結論（先に全体像）

- **方針：ネイティブアプリ申請が本命**（App Store / Google Play）。Web版は補助（ランディング＋Web契約導線）。
  - 理由：フィットネス/ウェルネスは「毎日開く習慣」が命。**プッシュ通知・ホーム画面アイコン・ストアでの発見性**はWebにない強み。課金アプリの信頼も「ストアにある」ことで高まる。revo の設計は完全にモバイルネイティブ（縦画面・タップ操作）。
  - Web版（Vercel）はランディングページとして、また「Web契約ならストア手数料なし」の導線として後から用意。捨てずに補助として活かす。
- **Mac は買わなくてよい。** React Native + **Expo（EAS Build）** を使えば、**iOS アプリも Windows からクラウドでビルド・申請できる**。
- 既存実装が **React（RevoApp.tsx）** なので、React Native への移行は比較的スムーズ。
- **1つのコードベースで iOS / Android 両対応**。revo のマルチアプリ展開（fit/life/running/world）にも向く。

技術構成の推奨：
```
モバイル：React Native + Expo（EAS Build でクラウドビルド → App Store / Google Play）
Web    ：Next.js（既存 page.tsx がベース）→ Vercel
バックエンド：Supabase（既存 schema.sql がベース）
決済   ：モバイル＝App内課金（StoreKit / Google Play Billing）／Web＝Stripe
コード置き場：GitHub
```

## 2. Windows で iOS アプリを出す仕組み（重要）

通常 iOS アプリのビルドには Mac（Xcode）が必要だが、**Expo EAS Build** はクラウド上の Mac でビルドを代行してくれる。つまり：

```
Windows でコードを書く
   ↓ git push
GitHub
   ↓ eas build（コマンド1つ）
Expo のクラウド（Mac環境）でビルド
   ↓
.ipa（iOS）/ .aab（Android）が生成される
   ↓ eas submit
App Store Connect / Google Play に自動アップロード
```

- **Mac 不要。** Windows のコマンドラインから `eas build` `eas submit` で完結。
- Expo は無料枠あり（ビルド回数に制限）。本格運用は有料プラン（月額）。
- 実機テストは Expo Go アプリ（自分のiPhone/Android）で確認できる。

15. アプリ説明文・キーワード・年齢レーティング（多言語）
16. プライバシーポリシー・特商法表記（実際の会社情報で。`legal_*` がベース）
17. プライバシー「栄養ラベル」（データ収集の申告・App Store必須）

### フェーズ6：ビルド・提出
18. `eas build`（iOS/Android）
19. `eas submit`（各ストアへ）
20. 審査対応 → 公開

## 4. ストア課金と価格戦略（今のうちに意識）

- **ストア手数料**：App Store / Google Play はデジタルサブスクに 15〜30% の手数料。
  - 小規模事業者プログラムなら 15%（年間売上100万ドル以下）。revo は当面これに該当する見込み。
- **手取りの例**（横断¥1,960/月・手数料15%の場合）：
  - 額面 ¥1,960 → 手取り 約¥1,666（¥294がストア手数料）
- **対策の選択肢**：
  - (a) ストア価格をそのままに、手取り減を許容（シンプル）
  - (b) Web（Stripe）契約に誘導し手数料を回避（※ストアのルールで「アプリ内からWeb決済へ露骨に誘導」は制限あり。地域差あり・要確認）
  - (c) ストア価格を少し上げて手数料分を吸収（ユーザー負担増）
- **推奨**：まずは (a) でシンプルに出し、規模が出たら (b)(c) を検討。**価格の見た目（¥1,960等）は維持**しつつ、手取りは内部で把握（管理システムの売上画面で手数料込みの実収益を見られるようにすると良い）。

## 8. 【方針転換】PWA（Web版＋スマホ版）を本命にする

**背景**：資金面・早期リリースを優先し、ネイティブ申請（App Store/Play）から **PWA** に方針転換。Stripe課金は維持（変更不要）。

### PWA とは
- Web技術で作りつつ、**ホーム画面にアイコン追加**でき、見た目・操作はアプリと同じ。
- **App Store/Play の審査が不要**（早期リリース）／**ストア手数料 15〜30% がかからない**（Stripe直課金のまま）。
- **Mac 不要・WSL 不要**（開発環境の仮想マシン問題も回避）。
- 既存の HTML モック（Web/Next.js ベース）がそのまま活きる。

### PWA 実装の要点
- `manifest.json`（アイコン・名前・テーマ色 #E3C56A・表示モード standalone）。
- Service Worker（オフライン対応・キャッシュ・「ホームに追加」を成立させる）。
- HTTPS 必須（Vercel は標準で HTTPS）。
- iOS Safari の「ホーム画面に追加」対応（Apple 独自の meta タグ等）。※ iOS PWA はプッシュ通知等に一部制約あり（要確認）。
- レスポンシブ（スマホ縦画面が主・PCも見られる）。

### 技術構成（PWA版）
```
フロント：Next.js（PWA対応）→ Vercel にデプロイ
バックエンド：Supabase（DB・認証）
決済：Stripe（Web直課金・手数料回避）
コード：GitHub
```

## 9. 【必須】商用リリースのセキュリティ精査（実装時チェックリスト）

お金と個人情報を扱う商用サービスの必須事項。実装時に必ず満たす。

### 認証・JWT（2026 ベストプラクティス）
- [ ] **アクセストークンは15分以内で失効**。長いセッションは**リフレッシュトークンのローテーション**。
- [ ] トークンは **localStorage/sessionStorage に置かない**。**HttpOnly・Secure・SameSite=Strict クッキー**に保存（XSSで盗まれない）。
  - 推奨：アクセストークンはメモリ、リフレッシュトークンは HttpOnly クッキー（ハイブリッド）。
- [ ] 署名アルゴリズムを**明示的に許可リスト指定**（`alg:none` を絶対に許さない＝改ざん攻撃防止）。
- [ ] JWT ペイロードに**パスワード・クレカ・個人情報を入れない**（Base64なので誰でも中身が見える）。入れるのは user ID・role・iss・aud・exp・jti のみ。
- [ ] `iss`（発行者）・`aud`（対象）・`exp`（失効）を**必ず検証**。
- [ ] トークンを**ログに出さない**（サーバーログ・エラー通知・分析ツール）。
- [ ] 認証は**自前実装せず、実績あるライブラリ/サービス**を使う（Supabase Auth 等）。自前でJWTを組むと事故りやすい。

### 不正利用防止
- [ ] **HTTPS 強制**（中間者攻撃対策・Vercelは標準）。
- [ ] **リフレッシュトークンのファミリー管理**：使い回し検知でファミリー全体を無効化。
- [ ] **レート制限**（ログイン試行・API呼び出しの回数制限＝総当たり攻撃・乱用対策）。
- [ ] **Stripe Radar**（不正決済検知）を有効化。
- [ ] 入力値の検証・サニタイズ（SQLインジェクション・XSS対策）。Supabase の RLS（Row Level Security）で「自分のデータしか見えない」を強制。
- [ ] パスワードは **bcrypt 等でハッシュ化**（平文保存しない）。
- [ ] 管理システム（revo admin）は**厳格なアクセス制御**（運営者のみ・多要素認証推奨）。

### 決済（Stripe）
- [ ] **Webhook 署名検証**（Stripeからの通知が本物か確認）。
- [ ] 冪等性キー（二重課金防止）。
- [ ] 金額・プランは**サーバー側で決定**（クライアントから送られた金額を信用しない）。
- [ ] 返金・解約・日割りのロジックをテスト（14日返金・前受金）。

## 10. 【必須】課金・コストの暴走防止（多額請求を防ぐ）

「知らないうちに多額の請求」を防ぐため、各サービスで**上限・アラート**を必ず設定する。
※ 設定場所・名称は各サービスで変わるため、実装時に公式ダッシュボードで最新を確認すること。

- **Vercel**：Spend Management（支出上限・使用量アラート）。無料枠を超える前に通知/停止。
- **Supabase**：使用量アラート・Spend Cap（無料枠/有料枠の上限）。
- **Stripe**：それ自体は売上側だが、Radar（不正）・請求のテスト。返金の暴発に注意。
- **もし AI API を使う場合（例：商品広告のAI生成）**：**必ず使用上限（Usage Limit）とアラートを設定**。API キーは絶対にフロントに置かない（サーバー経由）。従量課金は青天井になり得るので上限必須。
- **ドメイン・その他**：自動更新の請求日を把握。
- **共通の考え方**：
  - すべての従量課金サービスに**ハード上限（超えたら止まる）**か、最低でも**アラート**を設定。
  - 予算を月◯円と決め、それを超えたら通知が来る状態にする。
  - 無料枠の範囲を把握し、有料に移る境界を知っておく。
  - リリース初期は**小さく始める**（アクセスが少ないうちは無料枠で足りることが多い）。

## 12. 【実装注意】PWA の通知（プッシュ通知）— iOS制約と許可のタイミング

revo は「毎日開く習慣」が命。通知は重要だが、PWA（特に iOS）には制約があるため、実装時に必ず押さえる。

### iOS の Web Push 制約（2026時点・要・最新確認）
- **iOS 16.4以降**で対応（2023年3月〜）。2026年初時点で iPhone の95%以上が対象なので実用可能。
- **ホーム画面に追加した PWA でのみ通知が使える**。Safari のタブで開いているだけ、または他アプリ内ブラウザ（WKWebView）では通知不可。
  - → だから「ホーム画面に追加の案内（addhome 画面）」が通知の前提。順序が重要：**まずホーム追加 → 通知許可**。
- **自動インストールプロンプトが無い**（Android と違い、iOS は手動で「共有→ホーム画面に追加」が必要）。addhome 画面で手順を案内する設計はこのため。
- 通知の許可ダイアログは、**ユーザー操作（ボタンのタップ）に応じてのみ出せる**。
  - → notif_permission 画面の「通知をオンにする」ボタンを押した時に、Safari の許可ダイアログを出す。この設計で正しい。
- **バックグラウンド同期（Background Sync）は iOS では非対応**。通知の到達率は Android より低め。重要通知はメール等のフォールバックも用意。
- HTTPS 必須（Service Worker と Push にはセキュアなオリジンが必要・Vercel は標準対応）。

### 許可のタイミング（opt-in率を上げる設計）
- **いきなりブラウザの許可ダイアログを出さない**。まず revo 独自の「通知をオンにしませんか？」案内（notif_permission 画面）で価値を伝える＝pre-permission。
- **価値を感じた後に出す**：初回ログイン直後でなく、最初の記録・応援・メダルなど「通知が嬉しい体験」をした後が理想。
- ユーザーが案内で「オンにする」を選んだときだけ、実際のブラウザ許可ダイアログを出す。これで拒否（一度拒否されると再要求が難しい）を減らす。
- 「あとで」を残し、後から設定（notif_settings）でオンにできる導線を保つ。

### 実装フロー（通知まわり）
```
welcome → signup/login →（アプリ体験）→ addhome（ホーム画面に追加）
  → notif_permission（価値を伝える pre-permission）
  →「オンにする」タップ → ブラウザ許可ダイアログ → 許可
  → 以降 notif_settings で種類ごとに ON/OFF
```

### 実装手段
- Web Push は自前実装より、**プッシュ通知サービス（OneSignal / MagicBell / Pushwoosh 等）や Supabase + Web Push ライブラリ**を使うと、Service Worker・VAPIDキー・クロスブラウザ対応が楽。
- 通知の種類（応援・フォロー・メダル・記録リマインド・レシピ・停滞期の励まし）は notif_settings のトグルと対応させる。
- ※ iOS PWA の通知仕様は変化しうるので、実装時に Apple Developer の最新ドキュメントを確認。

## 13. 【最重要・実装注意】初期状態（数値ゼロ／エンプティステート）の扱い

**新規ユーザーは、記録がまだ何も無い「ゼロの状態」でアプリを開く。** ここを正しく設計しないと、初回体験が残念になり、離脱につながる。

### 大前提：モックのサンプル数値は「仮」
- 本フォルダのモック画面には、見栄えのためにサンプル数値（体重、消費カロリー、再生数、ランキング等）がハードコードされている。
- **これらは実装時に必ず「データベース（Supabase）から取得」に置き換える。**
- 新規ユーザーは DB にデータが無い → 自動的にゼロ／空で表示される。**サンプル数値をそのまま出してはいけない**（他人の数字が見えるような誤解を与える）。

### エンプティステート（空の状態）を必ず設計する
数値がゼロ・リストが空のとき、ただ「0」や空白を見せるのでなく、前向きに次の行動を促す。

| 画面 | ゼロ/空のとき | 出すべきもの（例） |
|---|---|---|
| ホーム/記録 | 記録なし | 「最初の記録をつけてみましょう」＋記録ボタン |
| 分析/グラフ | データなし | 平らなグラフでなく「記録がたまると、ここに推移が出ます」 |
| ランキング | 参加なし | 「今日から始めて、ランキングに載りましょう」 |

### 初期値の方針
- 数値項目（体重・目標など）は、オンボーディング（goal_setup）で本人に入力してもらう。入力前は空欄またはプレースホルダ。
- 集計値（消費カロリー・連続日数・スコア）は 0 から始まる。0 を恥ずかしく見せず、「これから増える」余地として見せる。
- ランキング・メダル・称号は「未獲得」状態を用意し、獲得条件をヒントとして見せる（次の行動の動機になる）。

### テスト時の必須確認
- **まっさらな新規アカウントで全画面を開き、サンプル数値が残っていないか**を必ず確認する（テストデータの消し忘れは、実際によくある事故）。
